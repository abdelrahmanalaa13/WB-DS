M3 column widths are percentage-based, not fixed. This allows content to flex within breakpoint ranges. Gutters and margins grow at each breakpoint to prevent content from feeling cramped on larger screens.

Apple's Approach: No Grid, Constraint-Based

Apple does not prescribe a column grid. Instead, they use Auto Layout constraints with leading and trailing anchors. Content width is governed by readability margins (max ~700pt on iPad) rather than column counts. The key measurement: minimum 44pt touch targets and system-defined safe area insets (status bar, home indicator, notch).

DS Application: For our Angular ERP, use a 3-tier system: Compact (0-599px, 4 columns, 16px margins), Medium (600-1023px, 8 columns, 24px margins), Expanded (1024px+, 12 columns, 24-32px margins). This maps to M3's approach but extends the Expanded breakpoint down to 1024px (standard for enterprise desktop). All spacing uses our 4px base grid.

Adaptive Layout Patterns

All major design systems converge on three canonical layout patterns that adapt between mobile and desktop:

List-Detail (Master-Detail)

Compact: Full-screen list. Tap navigates to full-screen detail. Back button returns.

Expanded: Side-by-side. List on the left (or start), detail on the right (or end). Selection highlights in list, detail updates in place.

ERP use: Conversation inbox, contact list, ticket management, settings pages.

Supporting Pane

Compact: Main content full-screen. Supporting pane accessed via bottom sheet or modal.

Expanded: Main content + persistent side panel. Panel can be collapsible.

ERP use: AI chat panel alongside workspace, property inspector, filter panels.

Feed / Content Grid

Compact: Single-column vertical scroll. Cards stack.

Expanded: Multi-column grid (2-3 columns). Cards fill available width.

ERP use: Dashboard widgets, analytics cards, notification feed, activity timeline.

Content Density

Apple's Approach

Apple uses readability margins that automatically constrain text width on large screens. On iPad in landscape, body text does not span the full width. Instead, it stays within ~700pt, centered. This prevents long line lengths that reduce readability.

Apple also reduces information density on mobile: fewer columns in tables, card-based views instead of rows, progressive disclosure via drill-down navigation.

M3's Approach

M3 defines maximum content width containers. On expanded screens, content does not stretch infinitely. A max-width container (typically 1040-1280dp) centers content with auto margins. This creates breathing room on ultra-wide screens.

For data-dense views (tables, dashboards), M3 recommends horizontal scrolling for columns that do not fit, rather than cramming all columns into the viewport.

RTL / Bidirectional: Overview

Right-to-left (RTL) layout support is a primary requirement for our DS: WideBot serves Arabic-speaking government and enterprise clients, Hulul serves Arabic-speaking SMBs. Arabic is the primary UI language, not an afterthought localization.

The Fundamental Rule

In RTL mode, the entire layout mirrors horizontally. What was on the left moves to the right. What was on the right moves to the left. Reading flow reverses: right-to-left, top-to-bottom. Navigation that went "forward to the right" now goes "forward to the left." The back arrow points right instead of left.

This is not a CSS transform or visual hack. It is a structural reversal of the layout's inline axis. The block axis (vertical) is unchanged.

Critical for our DS: Data tables with numeric columns, charts/dashboards, and phone/ID number fields must NOT mirror. But table row layout (icon, name, actions) DOES mirror. This means a data table mirrors its row structure but keeps number alignment and chart axes in LTR. Get this wrong and Arabic users will report the interface as broken.

Icon Mirroring Decision Tree

The 4 Categories (from Material Design)

1. Always mirror: Icons representing directionality in the UI. Back arrow, forward arrow, navigation arrows, text alignment icons, list indent/outdent, send button, reply, undo/redo, bullet list.

2. Never mirror: Icons representing physical objects (camera, headphones, key), universal symbols (checkmark, X close, plus, minus), media controls (play, pause, skip), clocks and circular indicators.

3. Mirror with caution: Icons with subtle directional meaning. A speech bubble with tail on the left might mirror. A pencil icon angled to suggest right-handed writing might need a left-angled variant for RTL. Evaluate case by case.

4. Provide locale-specific variants: Some icons need entirely different designs for RTL. The "text align left" icon becomes "text align right" in RTL, but it is not the same icon mirrored; it is a different concept. SF Symbols provides locale-specific variants for Arabic, Hebrew, and Devanagari.

DS Application: We use HugeIcons. We need to audit the full icon set and tag each icon as: mirror: always, mirror: never, or mirror: locale-variant. This tag should be in our AI Component Registry so generative agents know which icons to flip in RTL layouts. The DS_RULES.md already mandates dir="rtl" support. This icon audit is the missing piece.

Bidirectional Text Handling

The Challenge

Arabic text mixed with English words, brand names, URLs, or numbers creates bidirectional (bidi) content. A single sentence might contain:

مرحبا بك في WideBot! رقم الطلب هو 12345

In this example, "WideBot" and "12345" are LTR segments embedded in RTL text. The Unicode Bidirectional Algorithm (UBA) handles most cases automatically, but edge cases require explicit direction markers.

Rules for Our DS

1. Phone numbers: Wrap with U+202A (LRE) before and U+202C (PDF) after, or use <bdo dir="ltr"> in HTML.

2. Brand names in Arabic context: Use <bdi> (bidirectional isolate) element to prevent the brand name from affecting surrounding text direction.

3. URLs and email addresses: Always LTR. Use dir="ltr" on the container element.

4. Mixed number + Arabic text: UBA handles this correctly in most cases. Only intervene if the display is visually wrong.

5. Text input fields: Set dir="auto" on inputs so the direction is determined by the first strong character the user types. An Arabic user typing a phone number should see LTR input; typing Arabic text should see RTL input.

CSS Logical Properties: The RTL Foundation

CSS Logical Properties are the single most important technology for RTL support on the web. They replace physical directions (left/right/top/bottom) with logical ones (inline-start/inline-end/block-start/block-end) that automatically flip in RTL.

/* WRONG: Physical properties - breaks in RTL */ .card { margin-left: 16px; padding-right: 24px; border-left: 3px solid #4ade80; text-align: left; float: left; } /* CORRECT: Logical properties - works in both LTR and RTL */ .card { margin-inline-start: 16px; padding-inline-end: 24px; border-inline-start: 3px solid #4ade80; text-align: start; float: inline-start; }

Important distinction: The visual element can be smaller than the touch target. A 32px icon button can have a 48px touch target by adding transparent padding. This is how Apple achieves compact visual density while maintaining accessible tap areas. Material calls this the "touch target area" which extends beyond the visual bounds.

DS Application: Our DS should mandate 48px minimum touch targets (matching Material, exceeding Apple and WCAG AA). For WideBot (AAA), use 48px targets with 8px spacing. For compact components (data table action icons, chip close buttons), the visual element can be 24-32px but the touch area must extend to 48px minimum. Document this as: min-height: 48px; min-width: 48px on all interactive elements, with padding bridging the gap between visual size and touch size.

Thumb Zone and Reachability

Research Findings

75% of mobile interactions are thumb-driven. 49% of users hold the phone one-handed, 36% cradle it, 10% use two thumbs. The "thumb zone" is not fixed; users shift their grip based on context. But the natural resting position of the thumb creates three zones:

Easy zone (green): Lower-center and lower-inline-start of the screen. This is where primary actions should live.

Stretch zone (yellow): Middle of the screen and lower-inline-end. Reachable with some effort.

Hard zone (red): Top corners. Requires the user to shift their grip or use the other hand.

DS Application: This directly validates our mobile adaptation patterns: bottom navigation for primary nav (easy zone), bottom sheets for contextual actions (easy zone), FAB at bottom-inline-end (easy zone). Search and settings in the top bar (stretch/hard zone) are acceptable because they are less frequent actions. The key rule: frequently-used actions at the bottom, infrequently-used actions at the top.

Gesture Design

Standard Gestures

Tap: Primary action. Equivalent to click.

Long press: Context menu. Equivalent to right-click.

Swipe horizontal: Reveal actions (delete, archive) on list items. Navigate between tabs/pages.

Swipe down: Pull-to-refresh. Dismiss sheet/modal.

Swipe from edge: Back navigation (iOS). Open drawer (Android).

Pinch: Zoom. Used for maps, images, data visualization.

Two-finger drag: Pan within zoomed content.

Gesture RTL Considerations

Swipe to reveal actions: In LTR, swipe left reveals actions on the right. In RTL, swipe right reveals actions on the left. The gesture direction mirrors.

Edge swipe for back: In LTR (iOS), swipe from left edge to go back. In RTL, swipe from right edge. Apple handles this automatically. Angular apps must implement this explicitly.

Swipe between pages: In LTR, swipe left = next page. In RTL, swipe right = next page. The direction reverses because "forward" reverses.

Pull-to-refresh: Does NOT change in RTL. Vertical gestures are unaffected.

Actionable Takeaways for Our DS

1. Three-tier responsive grid. Compact (0-599px, 4 columns, 16px margins), Medium (600-1023px, 8 columns, 24px margins), Expanded (1024px+, 12 columns, 24-32px margins). Add the medium tier to our system; current DS only has mobile/desktop.

2. Navigation Rail for tablet. Define the navigation rail component: 72-80px wide, icons-only with optional labels below, persistent on left/right edge. This is the missing navigation tier between bottom tabs (mobile) and full sidebar (desktop).

3. 100% CSS logical properties. Enforce via linting: zero usage of margin-left, margin-right, padding-left, padding-right, left, right, text-align: left/right, float: left/right. All replaced with logical equivalents. This handles 80% of RTL automatically.

4. Icon mirroring registry. Audit all HugeIcons used in the DS and tag each with a mirror property: always (directional arrows, navigation), never (physical objects, checkmarks, media controls, charts), or locale-variant (text formatting icons). Store in the AI Component Registry so generative agents respect this.

5. dir="auto" on all text inputs. Every <input> and <textarea> must use dir="auto" so the input direction matches the first strong character typed. This handles the common case of Arabic users entering phone numbers (LTR) or Arabic text (RTL) in the same field.

6. Data table RTL rules. Row layout mirrors (icon on right, actions on left). Text columns use text-align: start. Number columns use dir="ltr" and text-align: end (visually on the left in RTL, but numbers read left-to-right). Sort indicator arrows mirror. Column order does NOT mirror (columns stay in their defined order).

7. Touch targets: 48px minimum, visual can be smaller. All interactive elements must have a 48x48px minimum touch area, but the visible element can be 24-32px with transparent padding extending the hit area. This matches Material Design and exceeds WCAG AA (24px). Document the pattern: min-height: 48px; min-width: 48px; display: flex; align-items: center; justify-content: center;

8. Bottom-zone priority for mobile actions. Primary actions at screen bottom (easy thumb zone). Secondary at middle. Tertiary at top. Validate every mobile layout against this rule. The FAB, bottom nav, and bottom sheets already follow this. Ensure new patterns do too.

9. Gesture direction reversal in RTL. Document in interaction contracts: swipe-to-reveal mirrors direction, edge-back-swipe mirrors, page-swipe mirrors, pull-to-refresh stays vertical (no change). This must be explicit for the Angular implementation.

10. Max content width for ultra-wide. Add a --content-max-width token (1280px) to prevent content from stretching across ultra-wide monitors. Center the content container with margin-inline: auto. This matches M3's approach and Apple's readability margins.
