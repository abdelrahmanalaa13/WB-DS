Apple HIG Deep Dive

What makes Apple design feel premium, accessible, and timeless. Distilled from the Human Interface Guidelines, WWDC sessions, and Apple's design philosophy into an indexed reference for our design system.

Apple's Core Design Principles

Apple's design philosophy has evolved through two eras. The original three principles (iOS 7 through iOS 25) and the new three (iOS 26 / Liquid Glass onward). Both remain relevant because the originals describe the "what" while the new ones describe the "how."

Clarity (Original)

Every element must be easy to understand. Clean text, sharp icons, strong visual hierarchy, focus on the most important elements. Clarity is not minimalism for its own sake. It is the elimination of ambiguity. A user should never wonder "what does this do?" or "where am I?"

Deference (Original)

The UI should step back and let user content take center stage. Interface elements should never compete with the user's content for attention. Translucent elements, subtle borders, and fluid motion help the chrome recede. The interface is a frame, not the painting.

Depth (Original)

Layers, shadows, and motion create spatial relationships that help users understand hierarchy and navigation. Where am I? Where did I come from? Where can I go? Depth answers these questions visually, without text.

Hierarchy (Liquid Glass)

Dynamic visual hierarchy where controls elevate and distinguish content beneath them. Unlike static approaches, hierarchy adapts based on user actions and context. Related actions group together. The interface prioritizes, conceals, or simplifies elements intelligently.

Harmony (Liquid Glass)

Software aligns with the physical and contextual realities of each device. Glass effects blend with content and hardware. Materials and shapes adapt to form factors. Touch interactions remain unobstructed. Apps reshape themselves contextually across multitasking.

Consistency (Liquid Glass)

Consistent design that continuously adapts across window sizes and displays. Not rigid uniformity but fluid coherence. The same component anatomy adapts meaningfully to device context without requiring complete redesigns. Single anatomy, platform expressions.

DS Application: Our DS already embodies Clarity (token-driven, documented), Deference (surface hierarchy, content-first), and Depth (elevation, materials). The Liquid Glass principles add: dynamic Hierarchy (our responsive system + progressive disclosure), Harmony (brand-specific radius/color tuning), and Consistency (multi-brand multi-platform from one component registry). These are not new concepts for us but new names for patterns we are building.

Liquid Glass (WWDC 2025)

Apple's most comprehensive design overhaul since iOS 7. Liquid Glass is a translucent material that behaves like physical glass: it refracts background content, responds to light with specular highlights, and dynamically adapts to context. It is rendered in real time on the GPU.

Visual Properties

Translucency: Background content refracts through glass surfaces, creating a physical connection between layers.

Specular highlights: Light sources move through space as users interact. Locking/unlocking causes light to travel around the material, defining silhouettes.

Color sampling: Glass tints itself based on background content, creating harmony without explicit color matching.

Real-time rendering: All effects are GPU-computed per frame, not baked textures.

Shape Archetypes

Fixed shapes: Constant corner radii independent of container size. Predictable geometry for layouts.

Capsules: Radius = half container height. Apple's primary approach for all interactive touch elements on iOS/iPadOS.

Concentric shapes: Inner radii = parent radius minus padding. Automatic nesting that prevents corners from appearing "pinched" or "flared."

What Other Design Systems Can Learn

1. Systematic shape language: 2-3 core shape archetypes create coherence across thousands of components while remaining mathematically simple.

2. Content-first layering: UI elements should enhance rather than obscure content. Layering systems benefit from clear material hierarchies.

3. Eliminate decorative elements: Remove manual emphasis, backgrounds, and borders when system appearance communicates hierarchy. Reduces complexity and improves maintainability.

4. Source-connected interactions: Action sheets spawning from triggering elements clarify causation and improve spatial understanding.

5. Platform-sensitive defaults: Same component anatomy adapts meaningfully to device context without complete redesigns.

DS Application: The concentric radius principle is directly applicable. When a card (--radius-medium: 16px for Hulul) contains a button, the button's radius should be card-radius minus card-padding. If card padding is 12px, button radius should be 4px, not the button's default radius. This prevents the "pinched corner" problem. We should add a --radius-inner token or computation rule to DS_RULES.md.

Why Apple Design Feels Good

Apple's design appeal is not one thing. It is the compounding effect of many small decisions that individually seem insignificant but collectively create an unmistakable quality signal. Here are the real mechanisms:

1. Optical Adjustments Over Mathematical Precision

Apple's SF Pro font uses optical sizing. At small sizes, letterforms are wider with more open apertures and heavier strokes. At large sizes, they become more refined with tighter spacing. This is not a single font at different sizes. It is a different design for each optical size. The result: text always looks "right" even though the metrics change.

Similarly, Apple adjusts tracking (letter-spacing) per size: -2.5% at 17pt body, -1.6% at 15pt, -0.61% at 13pt. Tighter tracking at larger sizes prevents letters from appearing to float apart. Looser tracking at small sizes prevents letters from colliding. This single detail is why Apple text feels more polished than most web typography.

2. Semantic Color Eliminates Decision Fatigue

Developers never pick colors. They pick roles. "label" not "#000000". "systemBackground" not "#FFFFFF". This means every color decision is already made optimally for light mode, dark mode, light high-contrast, and dark high-contrast. One semantic choice resolves to four optimized values. This eliminates the class of bugs where "dark mode looks wrong" because the system prevents it structurally.

3. Spring Animation as Default, Not Option

Apple replaced ease-based curves with spring physics as the default animation model. Springs are defined by mass, stiffness, and damping rather than duration and bezier curves. The key advantage: springs naturally absorb gesture velocity. When a user flicks a sheet, the spring picks up the gesture velocity and decelerates naturally. Ease-based animations cannot do this; they have a fixed duration regardless of initial velocity. This is why Apple interactions feel "connected to your finger" and most web UIs feel "separate from your input."

4. Materials Create Spatial Context

When a modal appears over content, the background is not just dimmed. It is blurred through a material layer that samples the underlying colors and creates a translucent surface. This does two things: it communicates depth (the modal is "above" the content) and it maintains visual connection (you can still see the content beneath, ghosted). This is why Apple modals feel like they are floating in space rather than pasted on top.

5. Consistency Through Constraint

Apple provides exactly 11 text styles, 9 system tint colors, 6 gray levels, and 3 background hierarchy levels. Not more. This extreme constraint means that any two iOS apps that follow HIG look harmonious together. Consistency is not achieved by guidelines. It is achieved by limiting the palette so thoroughly that inconsistency becomes structurally impossible.

Color: Semantic Architecture

Apple's color system is built on semantic naming. Colors are defined by their purpose, not their appearance. This single decision is the foundation of their entire theming and accessibility system.

The Four-Variant Model

Every color in iOS resolves to four values: light mode, dark mode, light mode + high contrast, dark mode + high contrast. When a developer uses .label, the system automatically selects from these four values based on the user's settings. This is not opt-in. It is the only way to use color correctly.

This prevents dark mode colors from looking washed out. The shift is subtle (3-5% luminance increase) but enough to maintain vibrancy against dark backgrounds. This matches the M3 tone-shift approach (tone 40 light to tone 80 dark) and our own approach of using lighter, brighter variants in dark mode.

Color Adaptation: Light / Dark / High Contrast

The Adaptation Matrix

Apple's color system adapts across two axes: appearance (light/dark) and accessibility (standard/increased contrast). This creates a 2x2 matrix where every color has four resolved values.

StandardIncreased Contrast
LightBase valueDarker, more saturated
DarkLighter, slightly more saturatedEven lighter, maximum contrast

When users enable "Increase Contrast" in Accessibility settings, colors shift to provide stronger differentiation. Borders become more visible. Fills become more opaque. Text colors become more saturated.

DS Application: We currently support a 2x2 matrix (2 brands x 2 themes). Apple's model suggests a 2x2x2 matrix (2 brands x 2 themes x 2 contrast levels). For WideBot (AAA), increased contrast mode would push to even higher ratios. For Hulul (AA), increased contrast mode would push from AA to AAA. This is a future enhancement worth planning for in our token architecture. The globals.css structure already supports it through nested data attributes.

Color Usage Rules

Apple's Color Rules

Use color sparingly. If everything is colorful, nothing stands out. Color should direct attention, not decorate.

Use complementary colors across light/dark. Not the same color dimmed. A color rebalanced for the new context.

Test under all appearances. Colors must work in light, dark, and both high-contrast modes.

Never rely on color alone. Always pair with shape, icon, or text label for accessibility.

Use semantic names. "label" not "black." "systemBackground" not "white." The name encodes purpose, not appearance.

Gray Scale as Information Architecture

Apple provides 6 system grays. These are not decoration. They are structural:

Gray 1 (#8E8E93 / #8E8E93): Disabled states, inactive icons.

Gray 2 (#AEAEB2 / #636366): Secondary borders, dividers.

Gray 3 (#C7C7CC / #48484A): Tertiary surfaces.

Gray 4 (#D1D1D6 / #3A3A3C): Subtle backgrounds.

Gray 5 (#E5E5EA / #2C2C2E): Grouped list backgrounds.

Gray 6 (#F2F2F7 / #1C1C1E): Page-level backgrounds.

Note how in dark mode, the grays INVERT their order. Gray 6 (lightest in light) becomes the darkest usable surface in dark mode. The semantic meaning stays constant even as the values flip.

Typography: The iOS Type Scale

Apple's type scale is not based on a mathematical ratio like modular scales. It is hand-tuned for each text style to optimize for readability at that specific size and weight combination. The sizes are pragmatic, not algorithmic.

Critical insight: Body and Headline share the same 17pt size. The only difference is weight (Regular vs Semibold). This is a deliberate design decision. It means emphasized text does not disrupt the vertical rhythm of body text. The line height stays constant. Only the visual weight changes. This is far more elegant than making headlines larger, which forces different line heights and breaks the flow.

Negative tracking at body sizes, positive at display sizes. At 17pt, Apple uses -0.43px tracking (letters slightly closer together). At 34pt, tracking is positive (letters slightly further apart). This inversion is counterintuitive but optically correct. At small sizes, tighter tracking improves word-shape recognition. At large sizes, looser tracking prevents the "wall of text" feeling and adds elegance.

SF Pro: The Font System

Font Variants

SF Pro Display: Optimized for sizes 20pt and above. Thinner strokes, tighter metrics, refined details.

SF Pro Text: Optimized for sizes below 20pt. Wider letterforms, more open apertures, heavier strokes for legibility.

SF Pro Rounded: Same metrics as SF Pro but with rounded stroke terminals. Used for playful/friendly contexts (Shortcuts, Activity rings).

SF Mono: Monospaced variant. Used for code, data tables where column alignment matters.

New York: Serif companion. Used for long-form reading (Books, News). Provides contrast when paired with SF Pro.

9 Weights

Ultralight, Thin, Light, Regular, Medium, Semibold, Bold, Heavy, Black.

Apple's practical usage:

Regular (400): Body text, most UI.

Medium (500): Subtle emphasis, navigation items.

Semibold (600): Headlines, buttons, emphasized labels.

Bold (700): Large titles, hero text.

Heavy/Black: Marketing only. Never in product UI.

Ultralight/Thin/Light: Decorative large display text only. Never for functional UI.

DS Application: Apple's 4-weight practical range (Regular, Medium, Semibold, Bold) maps well to our needs. For the ERP: Regular for body/data, Medium for labels and navigation, Semibold for buttons and emphasis, Bold for page titles. The optical size concept (Display vs Text) is handled by Inter's variable font optical sizing. We should ensure our Inter usage leverages optical sizing rather than using a single static weight file.

Dynamic Type: Scaling for Accessibility

Dynamic Type is Apple's system for scaling all text based on user preference. It ensures that users who need larger text get it, and the interface adapts gracefully. The system defines 12 size categories from xSmall to AX5 (accessibility extra extra extra large).

How Body Text Scales

At the default "Large" setting, body is 17pt. At "Extra Small," it drops to 14pt. At the largest accessibility size (AX5), it grows to 53pt. The scale is not linear. It uses a curve that grows faster at accessibility sizes to ensure readability for users with severe vision impairments.

Key rule: Text styles do not all scale at the same rate. Large Title scales from 31pt (xS) to 40pt (XXXL) but does NOT scale into accessibility sizes. Body text DOES scale into accessibility sizes. This prevents Large Title from becoming absurdly large while ensuring body text remains readable at any preference.

DS Application: For web/Angular, Dynamic Type maps to CSS clamp() with rem-based scaling. Our DS should define a type scale that responds to the browser's font-size setting (which users can adjust in OS settings). The formula: font-size: clamp(min, preferred, max) where preferred is rem-based. This gives us the same accessibility benefit as Dynamic Type on the web.

Typography Rules

Apple's Typography Rules

Use built-in text styles. Don't invent new sizes. Use the 11 provided styles.

Use Dynamic Type. Never hard-code font sizes. Always allow scaling.

Prioritize readability. Use minimum 11pt for any functional text. Decorative text can go smaller.

Maintain line length. Optimal reading width is 50-70 characters per line. Use layout constraints to prevent lines from becoming too wide.

Use weight for hierarchy, not size alone. Headline (17pt Semibold) and Body (17pt Regular) are the same size. Weight creates the distinction.

Shape Language

Apple's Shape Philosophy

Apple uses rounded rectangles as the universal shape primitive. Not circles, not sharp rectangles. The "squircle" (continuous-curvature rounded rectangle) is Apple's signature shape. Unlike CSS border-radius which creates circular arc corners, Apple's curves use a superellipse formula that creates smoother transitions between the straight edges and curves.

The squircle matters because: A standard border-radius creates a visible "kink" where the straight edge meets the curve. A superellipse/squircle has continuous curvature, meaning the curve starts before the mathematical corner point, creating a smoother visual transition. This is why Apple's app icons look "softer" than CSS-radius icons even at the same nominal radius.

Concentric Radius Rule

When elements are nested (button inside a card, icon inside a button), the inner element's radius must be calculated as:

Inner Radius = Outer Radius - Padding Between Them

If the outer card has radius 16px and padding 12px, the inner button should have radius 4px (16 - 12 = 4). If the inner element would have a negative radius, use 0 (sharp corner).

Why this matters: Without concentric radii, nested rounded rectangles create visual tension. The inner element's curve appears to "fight" the outer element's curve. With concentric radii, the curves flow in parallel, creating visual harmony.

DS Application: Add concentric radius calculation to our component system. For each container component (Card, Sheet, Dialog), document the expected inner radius for child interactive elements. Hulul card (radius 16px, padding 12px) produces inner radius 4px. WideBot card (radius 10px, padding 12px) produces inner radius 0px (sharp). This actually reinforces WideBot's sharper brand personality as a natural consequence of the math.

Materials and Vibrancy

Material Types

Materials in Apple's system are translucent layers that blur background content to create depth separation. They range from ultra-thin (barely visible, maximum background passthrough) to ultra-thick (heavy blur, near-opaque).

MaterialBlurOpacityUsage
Ultra ThinMinimal~10-20%Background overlays, subtle depth
ThinLight~30-40%Sidebar backgrounds, secondary surfaces
RegularMedium~50-60%Navigation bars, toolbars, sheets
ThickHeavy~70-80%Modal backgrounds, system surfaces
ChromeFull~90%+System chrome, tab bars

Vibrancy: Pulling Color Forward

Vibrancy is an effect applied to foreground content (text, icons, fills) that sits on top of a material. It blends the foreground with sampled background color, making the foreground feel like part of the environment rather than pasted on top.

Apple provides vibrancy levels for labels, fills, and separators, each with default (highest contrast), secondary, tertiary, and quaternary variants. The key principle: vibrancy increases the sense of depth without requiring explicit shadow or border. Content appears "in" the material rather than "on" it.

DS Application: For web, backdrop-filter: blur() approximates Apple's materials. Our command palette and AI suggestion panel should use: background: rgba(var(--surface-rgb), 0.7); backdrop-filter: blur(20px). This creates the material effect while maintaining text contrast. The --surface-dim token provides the fallback for browsers that do not support backdrop-filter.

Elevation and Shadows

Apple vs Material Design Shadows

Apple's shadow philosophy differs from Material Design. Material uses explicit shadow to communicate z-index (elevation 1 = small shadow, elevation 24 = large shadow). Apple uses materials and blur as the primary depth cue, with shadow as a secondary reinforcement.

Apple shadows are softer, more diffused, and lower contrast than Material shadows. A card in iOS has a barely perceptible shadow compared to a Material Design card. The depth is communicated primarily through the translucent material background, not the shadow.

The lesson: Excessive shadow looks dated. Modern depth comes from material contrast (slightly different background), subtle blur, and motion (elements that animate differently feel like they are at different depths).

Spring Animations

Why Springs, Not Ease Curves

Apple's motion system is built on spring physics. A spring animation is defined by three physical properties:

Mass: Heavier objects take longer to accelerate and decelerate. Controls the "weight" of the motion.

Stiffness: Stiffer springs move faster to their target. Controls the "snappiness."

Damping: Higher damping reduces oscillation (bouncing). Critically damped (damping ratio = 1.0) means no bounce, just smooth deceleration.

The critical advantage of springs over ease curves:

Springs accept initial velocity. When a user releases a gesture (like flicking a sheet), the spring animation starts with the gesture's velocity and decelerates naturally. An ease-based animation would ignore the gesture velocity and play at a fixed duration, creating a disconnect between the user's input and the result.

Springs have no fixed duration. They settle when the animation reaches equilibrium. Fast flicks settle in ~200ms. Slow drags settle in ~400ms. The duration is a consequence of physics, not an arbitrary value.

DS Application: Our current motion tokens use CSS cubic-bezier curves (--ease-standard, --ease-spring). CSS cannot do true spring physics. However, CSS spring() is proposed and may ship soon. In the meantime, our --ease-spring: cubic-bezier(0.175, 0.885, 0.32, 1.275) approximates a bouncy spring. For Angular, WAAPI (Web Animations API) supports spring-like easing through KeyframeEffect. For gesture-driven interactions (drag sheets, swipe actions), we should use JavaScript-driven springs rather than CSS transitions to capture gesture velocity.

Motion Design Principles

Apple's Motion Rules

Motion must reinforce spatial relationships. When you tap an item in a list and a detail view appears, the list item should expand into the detail view. The user sees the spatial connection: "the detail came FROM that item."

Motion must respect physics. Objects don't teleport. They accelerate, decelerate, and respond to forces. Motion that defies physical laws (like instant teleportation or linear constant-speed movement) feels wrong even if users can't articulate why.

Motion must be fast. Most iOS transitions are 250-350ms. Navigation pushes are ~350ms. Sheet presentations are ~300ms. Anything over 500ms feels sluggish.

Gesture-Driven Motion

Interaction continuity: When a user starts dragging a sheet, the sheet should track the finger 1:1. When released, the spring animation picks up the gesture velocity. The transition from direct manipulation to animation must be seamless.

Interruptibility: Every animation must be interruptible. If a user taps during a transition, the transition should immediately begin responding to the new input. Non-interruptible animations feel broken.

Rubber-banding: When a user drags beyond a boundary (like overscrolling), the element should resist with increasing force. This rubber-band effect communicates "you've reached the edge" without an abrupt stop.

Actionable Takeaways for Our DS

1. Adopt opacity-based label hierarchy. Instead of separate hex values for primary/secondary/tertiary text, use opacity on a base color: 100%, 60%, 30%, 18%. This automatically harmonizes text colors with any background and reduces token count. Matches Apple's label system.

2. Implement concentric radius rule. Add to DS_RULES.md: Inner interactive elements must use radius = parent radius - parent padding. Document per container component. This prevents the "pinched corner" artifact and matches Apple's Liquid Glass shape language.

3. Negative tracking at body sizes. Apply letter-spacing: -0.02em at 16-17px body sizes and letter-spacing: +0.01em at 28px+ display sizes. Matches Apple's tracking curve and improves word-shape recognition at reading sizes.

4. Weight-based hierarchy within same size. Apple uses Body (17pt Regular) and Headline (17pt Semibold) at the same size. Our DS should define emphasis through weight change, not size change, for inline hierarchy. This preserves vertical rhythm.

5. Material tokens for overlays. Define --material-thin, --material-regular, --material-thick as combinations of background opacity + backdrop-filter blur. Command palette = material-thin. Modal = material-regular. System overlay = material-thick.

6. Dark mode: shift hue AND luminance. Apple shifts colors by +1-3deg hue and +3-5% luminance for dark mode, not just lightening. Our dark mode tokens already do this (e.g., danger goes from #d13029 to #ff8383, shifting both hue and luminance). Validate this is consistent across all roles.

7. Four practical font weights only. Regular (body), Medium (labels), Semibold (buttons/emphasis), Bold (titles). Avoid Thin/Light/Heavy/Black in product UI. Reserve extremes for marketing only.

8. Animation interruptibility as a requirement. Add to interaction contracts: every animated transition must be interruptible by new user input. Non-interruptible animations are bugs, not features.

9. Plan for high-contrast mode. Design the token architecture to support a third axis (standard / increased contrast) beyond brand and theme. Even if not implemented in v1, the structure should accommodate it without refactoring.

10. Content-first layering. Apple's "deference" principle: UI chrome should never compete with content. In our ERP context: toolbars, navigation, and filters should use neutral/subdued surfaces. Only user content (data, conversations, documents) gets the full contrast range.
