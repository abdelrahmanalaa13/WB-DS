Architecture v3: M3 Foundation Strategy FINAL

Own the specification. Fork M3 tokens. Delegate implementation to framework adapters. Future-proof for Flutter, white-label, and direction changes.

The Five-Layer Ownership Model

Principle: You OWN layers 1-2. You FORK layer 3. You ADAPT layer 4. You IMPLEMENT layer 5. If M3 changes direction, only layers 3-5 are affected. Your intelligence and specifications are untouched.

YOU OWN Intelligence Layer (Knowledge Graph + Contracts + Rules + Agent Bus) Zero M3 dependency. Fully yours. Cannot be disrupted by Google.
Knowledge Graph
Queryable entity-relationship store. Impact analysis, overlap detection. Framework-agnostic. M3 changes cannot touch this.
Component Registry
YOUR component specs. Anatomy, slots, states, AI hints, interaction contracts. The registry describes WHAT, not HOW. Implementation is separate.
Rules Engine
WCAG AAA/AA per brand. RTL rules. Governance lifecycle. Touch model. These are YOUR business rules, independent of any design framework.
Agent Orchestration
ds-copilot, compliance, a11y, frontend, ai-agent. Coordination protocol, conflict resolution. Entirely your IP.
|
YOU OWN Specification Layer (Registry JSON + Patterns + Templates + Interactions) Framework-agnostic component definitions. Same spec generates Angular, Flutter, or any future target.
Component Specs
Properties defined as abstract types (not Angular signals, not Dart). Angular adapter maps to signals. Flutter adapter maps to widget props. Spec stays the same.
Pattern Compositions
AI assistant panel, data table, workflow approval. Defined as component graph (which components compose where). Layout is responsive CSS Grid, works on web and can map to Flutter Row/Column.
Interaction Contracts
Trigger, state machine, feedback, error behavior. Platform-neutral. Angular uses RxJS. Flutter uses streams. Contract is the same.
White-Label Config Schema
Enterprise customer picks source color. Dynamic Color generates palette. Config defines: brand name, source color, WCAG level, locale, RTL flag. That is the entire input.
|
FORK FROM M3 Token Layer (Forked M3 Tokens + Your Brand Overrides + DTCG Format) Clone material-foundation/material-tokens. Add your brands. Build with Style Dictionary v4.
Reference Tokens (from M3)
Cloned from material-tokens repo. Tonal palettes, type scale, motion curves, spring physics. DTCG format ($value, $type). This is your foundation. If M3 changes, you choose whether to pull updates.
System Tokens (your overrides)
Map reference tokens to semantic roles: primary, secondary, surface, error. Each brand (WideBot, Hulul, white-label) defines its own system token set. This is where differentiation happens.
Component Tokens (auto-cascaded)
Generated from system tokens via resolver. button-bg = system.primary. Change system.primary, every button updates. M3's cascade logic cloned into your build pipeline.
Dynamic Color Engine
M3's algorithmic palette generation. Input: one hex color. Output: 5 key colors, 5 tonal palettes, 50+ tokens with guaranteed AA contrast. Post-process for AAA (WideBot).
|
ADAPT Framework Adapter Layer (Tokens to Platform-Specific Output) Style Dictionary v4 transforms DTCG tokens to CSS vars, Dart constants, SCSS maps, TypeScript types.
Web Adapter (Angular)
DTCG tokens -> CSS custom properties (--ds-color-primary). SCSS 7-1 architecture. Brand-switchable via [data-brand]. Angular CDK for behaviors.
Mobile Adapter (Flutter)
DTCG tokens -> Dart constants (DsColor.primary). MaterialApp theme. ColorScheme.fromSeed() for Dynamic Color. Same source, different format.
Figma Adapter
DTCG tokens -> Figma Variables via figma-md-sync. Per-brand collections. Light/Dark modes. LTR+RTL frames with Arabic text.
White-Label Generator
Customer source color -> Dynamic Color -> DTCG tokens -> All adapters. One input, all platforms updated. Brand creation is automated.
|
IMPLEMENT Component Implementation (Angular 21 Now, Flutter Later) Components implement the spec using the adapted tokens. CDK for behavior. Your design language for visuals.
Angular 21 Components
Standalone, signal-based, OnPush. CDK for a11y + interactions. Styled with your CSS vars, not M3 defaults. Looks like Quo/Gemini, not Google.
AI-Native Components
Chat bubble, streaming text, assistant panel, tool results. No M3 equivalent exists. Built fully custom with CDK behaviors.
Flutter Components (Future)
Same spec, Dart implementation. Flutter M3 widgets as base where useful (ThemeData, ColorScheme). Custom widgets for AI/ERP.

M3 Direction Change Risk Mitigation

Threats (What Could Go Wrong)

M3 to M4 breaking change. Google could release M4 with incompatible token structure, like M2 to M3 transition that broke everything.
Angular Material direction shift. Angular team could deprioritize Material in favor of CDK-only approach (they hinted at this in 2025 blog posts).
Aesthetic shift. Google could adopt a drastically different visual direction (glassmorphism, neumorphism) that conflicts with enterprise readability.
Dynamic Color algorithm change. Palette generation could change, producing different colors from the same input, breaking white-label consistency.
DTCG format evolution. W3C could release a breaking update to the token format spec.

Shields (How We Are Protected)

Layers 1-2 have zero M3 dependency. Knowledge graph, registry, rules, agents, specs are entirely yours. M4 cannot touch them.
M3 tokens are FORKED, not linked. You control when/if to pull upstream changes. A breaking M4 release does not auto-break your system.
Token cascade logic is CLONED into your pipeline. Even if M3 changes its propagation mechanism, your Style Dictionary config is yours.
Dynamic Color is a BUILD step, not a runtime dependency. You run it once to generate palettes. White-label tokens are static after generation. Algorithm change does not affect existing brands.
Component implementations are YOUR code. Angular and Flutter components use your tokens, not mat-* components directly. Google's aesthetic changes do not propagate.
Worst case: replace Layer 3. If M3 becomes unusable, you replace the token layer only. Layers 1-2 (intelligence) and 4-5 (implementation) are unaffected. Estimated effort: 2-3 weeks.
Key Insight: The architecture is designed so that Google provides the SCIENCE (color theory, contrast math, motion physics, tonal palettes) and you provide the INTELLIGENCE (knowledge graph, governance, AI generation, multi-brand, RTL, learning). Science is universal and stable. If Google changes their science, the improvements likely benefit you. If they change their aesthetic opinions, you are insulated because your components are custom, not Material widgets.
